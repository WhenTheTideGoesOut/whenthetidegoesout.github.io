Approach to SQL Queries

In this post I want to address an approach on how to attempt SQL queries I find useful. In the previous posts we have looked at the setting up of a local SQL environent for you to practice on and then how SQL statements are structured and the order of operations. Please feel free to go back and have a quick read.

Now on to the issue at hand. How to approach an SQL query.

1) Start at the beginning.

This may sound a bit facetious but it is not. What I mean is, start with the problem you are trying to resolve. What is the question you are trying to ask of the DBMS.

Now the question or problem you are trying to resolve may originate from a colleague. In which case a meeting will be needed. Things to understand.

1. What is the question.
2. Why are you or why are they asking the question.
3. What is their expectation of the result.
4. What is the result going to be used for, what will it support.

While the first question is obvious, it is important and equally, the remaining three questions inform the first and can help refine and restructure the question to better resolve the actual problem.

2) Understand the Data

This may be a mute point given your experience and understanding of the data set in question. If you have been working with the database, day in day out for many years, then it is probably a given that you understand the data. We will progress any way.

When we say understand the data what we mean is to understand the following:

  1. The objective of the database at hand.
  2. The schema of the database. By this we mean how the different tables relate. The primary key field(s) and what each table represents.
  3. The Data Dictionary, once you understand the objective of the database and the schema or structure of it. It is important to understand the attribute within each of the tables. The rules they must follow etc.

This step could potentially come before the first in, start at the beginning. It could also be a given and not necessary for every problem.

The next step is to explore the dataset.

3. Explore the Data.

While at the outset it might seem that if you are working with the database day in day out and have many years experience, this may not be necessary. I believe however, that as the database changes and as the data within it changes this will still need to be completed.

The objective of this step is to help build an understanding that would allow you to build the right query to resolve the problem.

If we assume we are working within the one, known database, then commands such as:

  1. SHOW TABLES; will list out all of the tables within the database. You should be able to see the tables described in the schema document, ‘live’ within the database.
  2. DESCRIBE [table name]; will list out the field formats within the table. This should reflect to what is in the datadictionary.
  3. SHOW COLUMNS FROM [table name] will show the field names or column names from the table name supplied.
  4. SELECT COUNT (*) FROM [table name] will return the number of records in a given table.
  5. SELECT * FROM [table name] WHERE [col name] IS NULL OR [col-name] = ''; will return all the rows that have a null or emplty value in the field that was specified by [col name].

There are many ways to explore the data within the database but these are a good starting point. These can help with the next step which is to start to think about what the solution will look like.

4. Vizualize the Solution

Now that we have a firm understanding of the data and the problem at hand, it is time to visualize the solution.

What we mean by this, is really quite simple. Given our understanding of the data and our understanding of the problem, what will the solution look like.

What is the structure of the query, specifically:-

  1. What fields do we think we will be selecting?
  2. Which tables will we take the fields from?
  3. What conditions will be applied? and
  4. Will there be any aggregation?

It is through these steps and steps like these you will start to visulize or have an idea of what the solution will look. This leads perfectly on to step five, the last of the steps in our approach to the problem.

5. Code and Increment.

This step is really quite easy and should be taken as a work in progress.

In this step, start translating the answers from the questions in step four to SQL and running it.

Do not be discouraged about not getting it right first time. Run it, and re-run it. If you get errors, stop, think and google.

Wrap Up

This how I try and tackle my SQL problems. Some of my attempts are more successful than others. I try to approach it in this structured approach, in turth, I think, so long as there is structured approach to it you are increasing your chances of creating the right query.

Go forth and SQL.
